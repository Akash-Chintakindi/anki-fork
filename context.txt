1. ARCHITECTURE OVERVIEW (the 30-second model)
-------------------------------------------------------------------------------
Anki is ONE app split across three languages with a single, uniform boundary.

  Rust (rslib/)        = the brain. ALL real logic + the SQLite database.
                         Includes the spaced-repetition scheduler (FSRS + SM-2).
  Python (pylib/,      = the glue + desktop shell. pylib is a THIN facade that
          qt/aqt/)       proxies to Rust; qt/aqt is the PyQt window + webviews.
  TypeScript (ts/)     = the UI rendered inside embedded webviews (Svelte 5 +
                         SvelteKit), served to Qt by a local Flask server.
  Protobuf (proto/)    = the contract. Defines every backend method AND the
                         on-disk format of some collection items.

Key mental model:
  - pylib/anki is a facade: nearly every method builds a protobuf request,
    calls Rust, and parses the protobuf reply. Even SQL is shipped to Rust.
  - The desktop GUI is a STATE MACHINE driving three stacked webviews
    (top toolbar, shared central web area, bottom bar).
  - New UI is written in ts/ (SvelteKit) and copied into qt/_aqt/data/web/ at
    build time. (Note the underscore — see the gotcha in section 5.)

2. THE GOLDEN PATH — how one backend call flows across all 3 languages
-------------------------------------------------------------------------------
Frontend (TS)                Python (runtime intermediary)        Rust (logic+DB)
-------------                -----------------------------        ---------------
foo() in @generated/backend
  -> postProto("foo", msg)   HTTP POST /_anki/foo (binary body)
  (ts/lib/generated/post.ts)  -> qt/aqt/mediasrv.py dispatch
                                 (looks up name in exposed_backend_list)
                              -> col._backend.foo_raw(bytes)
                                 (pylib/anki/_backend.py _run_command)
                              -> _rsbridge.Backend.command(service, method, bytes)
                                 (pylib/rsbridge/lib.rs, PyO3, releases GIL)
                                                                   -> run_service_method
                                                                      (GENERATED dispatch)
                                                                   -> impl XService for
                                                                      Collection (HAND-WRITTEN,
                                                                      rslib/src/<area>/service.rs)
                                                                   -> col.transact(Op, |c| ...)
                                                                   -> col.storage.<query>()
                                                                      (SQL via include_str!)
                                                                   -> SQLite (rusqlite)
  <- typed object  <----------  <- bytes  <----------------------  <- proto-encoded bytes

The SAME (service, method) integer indices are derived from proto declaration
ORDER and are shared by Rust, Python, and TS with zero manual bookkeeping.
=> RULE: never reorder services/rpcs in proto; only APPEND. (See section 5.)

There is also a second, lighter channel for UI events only:
  pycmd / QWebChannel "bridge": JS bridgeCommand("...") -> Qt ->
  AnkiWebView.onBridgeCmd -> per-screen Python handler; Python pushes back with
  web.eval("jsFunc(...)"). Use for button clicks/save/close, NOT bulk data.

5. CROSS-LANGUAGE BOUNDARY RULES & GOTCHAS (read before editing)
-------------------------------------------------------------------------------
  - APPEND-ONLY PROTO. The (service, method) indices come from declaration order
    and are shared by all three languages. Reordering silently misaligns Rust/
    Python/TS. Only append new services/rpcs/messages.
  - qt/_aqt vs qt/aqt. Built web assets land in qt/_aqt/ (underscore — a GENERATED
    package). Editing anything under qt/_aqt is futile; it is overwritten each build.
    Source lives in ts/ and qt/aqt/.
  - GENERATED FILES are not in source and must not be hand-edited:
    _backend_generated.py, *_pb2.py/.pyi, hooks_gen.py, _fluent.py, buildinfo.py,
    _rsbridge.*, and out/ts/lib/generated/* (backend.ts, ftl.ts, anki/*_pb.*).
    They appear only after a successful build; expect IDE "cannot find module
    @generated/backend" until then.
  - TWO BACKEND CHANNELS: protobuf-over-HTTP (postProto -> /_anki/<method>, for
    structured data) vs pycmd/QWebChannel bridge (string/JSON, for UI events).
    Pick the right one (section 2).
  - WEB PAGE WIRING IS THREE LOCK-STEP EDITS for a backend-connected page:
    (1) the ts/routes folder, (2) is_sveltekit_page in mediasrv.py,
    (3) AnkiWebViewKind in the webview.py API-access list. Miss #3 and every RPC
    silently 403s.
  - DB ACCESS FROM UI must go through CollectionOp/QueryOp (never block the UI
    thread) and Rust mutations through col.transact(Op, ...) (or undo/caches break).
  - OLD WEBVIEW TARGET. The embedded Chromium is pinned low (chrome77/es2020);
    some modern JS libraries may break or down-level. Prefer the in-repo
    component/lib ecosystem.

7. RECOMMENDED TECH STACK FOR THE GMAT STUDY APP
===============================================================================
Founding principles (these drive every architectural choice below):

  PRINCIPLE 1: "Lessons create false confidence. Application creates competence."
  PRINCIPLE 2: "The problem is not content, motivation, or difficulty.
                The problem is learning structure."

The strategic insight: Anki is not just a convenient shell — its core value IS a
learning-structure engine (a scheduler that decides what you practice and when).
That is EXACTLY the problem Principle 2 names. So we do not bolt a quiz app onto
Anki; we extend Anki's structure engine into a GMAT competence engine.

RECOMMENDED STACK (reuse the existing Anki stack — that's the whole point of the fork):

  Learning-structure engine (the differentiator)  ......... RUST (rslib)
    - Extend rslib/src/scheduler/* so the default loop is adaptive, spaced,
      INTERLEAVED practice by topic and difficulty (queue/builder + states).
    - Track per-skill mastery in CardData JSON (no migration) and feed it back
      into scheduling (answering/ + StateContext).
    - Keep FSRS as the spacing backbone; layer GMAT-specific selection on top.
    => This is how we operationalize Principle 2.

  Content model (content is data, not code) ............... ANKI NOTETYPES
    - GMAT item types as notetypes: Problem Solving, Data Sufficiency, Reading
      Comprehension, Critical Reasoning, Sentence Correction.
      (AS BUILT — see section 8: 3 notetypes — GMAT PS, GMAT Verbal (CR+RC), and
      GMAT DI (DS / Two-Part / Multi-Source). Sentence Correction is OUT of scope;
      Data Sufficiency is a Data-Insights leaf, not its own notetype.)
    - Authored as fields + card templates; ships as data, versionable & importable.

  Orchestration + desktop shell .......................... PYTHON (pylib + qt/aqt)
    - New "GMAT" state/screen (section 4b), thin wrappers over col.* APIs,
      CollectionOp/QueryOp for all data access.

  UI (application-first, minimal passive reading) ........ TYPESCRIPT + SVELTE 5 /
                                                           SVELTEKIT
    - SvelteKit pages (section 4c) using the in-repo component library and SASS
      design tokens (dark mode for free).
    - DESIGN RULE from Principle 1: the home screen is a problem to solve, not a
      lesson to read. Every micro-lesson is immediately followed by application;
      explanations are revealed AFTER an attempt, never before.
    - Build the quiz/practice surface in the interactive editor-style Svelte
      pattern; do NOT fork the reviewer (it is tightly bound to the scheduler —
      instead drive it through the scheduler).

  Type-safe contract ..................................... PROTOBUF (proto/anki)
    - Add GMAT backend methods append-only (section 4d) when data must come from Rust.

  Content / cloud / auth (low-coupling, optional) ........ FIREBASE
    - A Firebase MCP is already configured in this workspace, making it the
      lowest-friction source for the GMAT question bank, user accounts, and
      cross-device progress. A SvelteKit +page.ts can fetch content directly from
      Firebase with ZERO Rust changes (ideal for the first milestone), while the
      scheduling/mastery state stays local in the Rust/SQLite collection.

  Dev workflow ........................................... ./run  (+ vite HMR :5173)
    - RELEASE=1 or ./tools/runopt for optimized builds; ./ninja check for tests.

SUGGESTED BUILD ORDER (de-risked, principle-aligned):
  1. GMAT notetypes (4a) + a SvelteKit practice page (4c) sourcing questions from
     Firebase or static JSON — an application-first loop with no Rust changes.
  2. Wire that loop to Anki's scheduler so spacing/interleaving is real (4e, tuning
     deckconfig first).
  3. Add custom mastery scheduling + topic search qualifiers (4e/4f) — the
     learning-structure differentiator.
  4. Add a mastery analytics dashboard (4g) and, only if needed, native backend
     methods/tables (4d/4h).

Net: content is the cheap part (Firebase/notetypes); the moat is the learning
STRUCTURE we build into Anki's Rust scheduler, surfaced through an
application-first Svelte UI — directly serving both founding principles.

    gmatwiz/lessons/ ................. authored micro-lessons per topic
                                         (topics_*.py, incl. topics_data_di.py).

8b. SECTION-AWARE TOPIC IDS (the key that ties everything together)
    Every card / lesson / plan / score keys off a leaf topic id of the form
        gmat::<section>::<category>::<leaf>
    e.g. gmat::quant::algebra::quadratics, gmat::verbal::cr::weaken,
         gmat::verbal::rc::main_idea, gmat::di::reasoning::data_sufficiency.
    The SECTION is the 2nd "::" segment; recover it with section_of() (taxonomy.py),
    _gmat_section_of_topic() (mediasrv.py), and gmat_section_of_topic() (rslib) —
    these three MUST agree. Taxonomy is defined once in gmatwiz/content/taxonomy.py:
      Quant  = 18 leaves (arithmetic 10 + algebra 8; PS only — NO geometry, NO DS)
      Verbal = 16 leaves (CR 9 + RC 7; NO Sentence Correction)
      DI     =  3 leaves (data_sufficiency, two_part_analysis, multi_source_reasoning)
      => 37 leaf topics total (ALL_TOPICS). Coverage % = distinct topics WITH cards
         over 37. A weighted keyword tagger (per-section rule sets) auto-classifies
         scraped/authored items to a leaf.

8c. NOTETYPES, DECKS, CONFIG KEYS (the data model)
    Notetypes (pylib/anki/gmatwiz.py):
      GMAT PS ...... Stem, OptionA-E, Correct, Explanation, Topic, Difficulty, Source.
      GMAT Verbal .. PS fields + Passage + PassageId (RC shares one Passage across
                     its cards via PassageId; the template renders the passage only
                     when present, so CR looks like PS).
      GMAT DI ...... identical fields to Verbal (Passage = a Multi-Source stimulus).
    Decks: GMAT::Quant, GMAT::Verbal, GMAT::DI (own preset, 9999/day limits).
    Per-section col.config keys (Quant keeps the ORIGINAL names):
      diagnosis: gmatDiagnosis / gmatDiagnosisVerbal / gmatDiagnosisDI
      plan:      gmatPlan      / gmatPlanVerbal      / gmatPlanDI
      profile:   gmatProfile (exam_date, days_per_week, target_score)
      content:   gmatContentVersion (bundled-import high-water mark)
      + the living-plan keys (gmatLearned, gmatQuizzes, gmatErrorLog, ...).
    Memory/Performance/Readiness are NOT config — they come from the Rust engine.

10. NEW INVARIANTS & GOTCHAS (respect these when touching GMATWiz)
-------------------------------------------------------------------------------
  - SCORING LOCKSTEP. rslib/src/gmatwiz.rs is the source of truth for Memory /
    Performance / Readiness; mediasrv.py mirrors its shape and api.ts mirrors its
    types. Change one -> change all three, or desktop / iOS / UI drift.
  - SECTION FROM TOPIC ID. Derive section from the 2nd "::" segment (8b), never
    guess from a deck/notetype alone; the three implementations must stay in sync.
  - IMPORT AFTER SYNC, THEN UPLOAD. Bundled-content import runs only AFTER the
    whole-collection sync settles (else colReplace clobbers it). Drive it from the
    client post-sync (gmatEnsureContent), NOT from render_gmat_screen.
  - CONTENT VERSION IS A HIGH-WATER MARK. Ship new bundled questions by bumping
    GMAT_BUNDLED_CONTENT_VERSION (qt/aqt/gmat.py); the import is idempotent, so it
    re-runs once and adds only new stems.
  - NEW GMAT ENDPOINTS need BOTH registration in post_handler_list AND (for POST) a
    whitelist entry in _check_dynamic_request_permissions (mediasrv.py).
  - REMOVE BEFORE DEPLOY: the daily-lock bypass password "1234"
    (STUDY_BYPASS_PASSWORD in ts/routes/gmat/GmatApp.svelte, used by lockedGate) is
    a dev-only affordance — strip it before any real release.
