===============================================================================
GMATWiz - AI EVAL + BASELINE  (Friday proof: "AI added and checked")
===============================================================================
What this proves (checklist items 3 + 4):
  3. An eval that runs BEFORE students see anything: accuracy + wrong-answer rate
     on a held-out set, with a documented cutoff.
  4. A side-by-side showing the AI beats a simpler method (keyword / vector search).

TASK EVALUATED: topic auto-tagging. Every question is routed to exactly one
taxonomy leaf topic; coverage %, mastery, and topic-aware scheduling all key off
that tag, so a wrong tag makes a student practice the wrong thing. The tagger is
therefore a gate the content pipeline runs before shipping questions.

HELD-OUT SET: gmatwiz/content/seed.json - 42 authored questions, each with a
human-assigned gold leaf topic. All three taggers are zero-shot (none is trained
on this set), so it is a valid held-out evaluation.

TAGGERS:
  - ai      : the app's model (OpenAI gpt-4.1-mini) via ai_ingest.OpenAIClient.
  - keyword : taxonomy keyword rules (ai_ingest.HeuristicClient) - simple baseline 1.
  - vector  : tf-idf cosine 1-NN over the set, leave-one-out - simple baseline 2.

CUTOFF (the ship gate): confidence >= 0.6 (ai_ingest.CONFIDENCE_THRESHOLD). An
item below the cutoff is flagged for human review instead of auto-shipping.
Metrics:
  accuracy          - exact leaf-topic match vs gold
  domain-acc        - correct top-level category (arithmetic vs algebra)
  coverage>=cut     - fraction confident enough to auto-ship
  acc@accepted      - accuracy among the items at/above the cutoff (what ships)
  conf-wrong        - share of ALL items that are >= cutoff AND wrong (a wrong tag
                      that would reach a student - the number the cutoff minimizes)

-------------------------------------------------------------------------------
REPRODUCE
-------------------------------------------------------------------------------
  cd gmatwiz/content

  # Baselines only (no key, offline) - keyword + vector:
  python3 eval_tagging.py

  # Full run incl. the AI tagger - uses the SAME key as the app (the Functions
  # secret). One line pulls it from Firebase and scores the AI row:
  export OPENAI_API_KEY="$(npx -y firebase-tools@latest functions:secrets:access OPENAI_API_KEY --project gmatwiz)"
  python3 eval_tagging.py            # writes gmatwiz/content/eval_report.json

  # Offline deterministic smoke test (AI = mock, random - not a real score):
  python3 eval_tagging.py --mock

Full machine-readable results (per-item predictions incl. confidences) are written
to gmatwiz/content/eval_report.json.

-------------------------------------------------------------------------------
RESULTS - BASELINES (real, this commit; gold=42, cutoff=0.6)
-------------------------------------------------------------------------------
metric                 keyword        vector
accuracy                 69.0%         35.7%
domain-acc               88.1%         78.6%
coverage>=cut            66.7%          0.0%
acc@accepted             96.4%           -
conf-wrong                2.4%          0.0%

Reading it: the keyword baseline is strong on this authored set (69% exact leaf,
96.4% among the items it is confident enough to ship, and only 2.4% confidently-
wrong). The tf-idf vector baseline is weak here (35.7% leaf) and never clears the
0.6 cutoff on such short math stems (0% coverage) - i.e. vector search alone is
not confident enough to auto-ship anything.

-------------------------------------------------------------------------------
RESULTS - FULL RUN with the AI tagger (real; OpenAI gpt-4.1-mini; gold=42, cut=0.6)
-------------------------------------------------------------------------------
metric                      ai       keyword        vector
accuracy                 81.0%         69.0%         35.7%
domain-acc               83.3%         88.1%         78.6%
coverage>=cut           100.0%         66.7%          0.0%
acc@accepted             81.0%         96.4%           -
conf-wrong               19.1%          2.4%          0.0%

RESULT: the AI tagger BEATS both baselines on accuracy (81.0% > 69.0% > 35.7%).
(eval_report.json records "ai_beats_baselines": true.)

Reading it (the honest nuance): the AI wins on raw leaf accuracy AND tags every
item confidently (100% coverage). The keyword baseline is lower-accuracy (69%) but
ABSTAINS on ~1/3 of items, so the tags it DOES ship are very clean (96.4%
acc@accepted, 2.4% confidently-wrong) - whereas the AI, being confident on
everything, ships more wrong tags (19.1% confidently-wrong). Practical takeaway
supported by these numbers: use the AI tagger for coverage/accuracy, but keep the
0.6 cutoff + human review on the confidently-wrong tail (and the keyword rules make
a good cheap cross-check). This is exactly the "runs before students see anything"
gate the cutoff exists for.

NOTE (honesty): the gold set is 42 items - a demonstrative held-out eval. Hand-
labeling ~50-100 more items from questions.json would tighten the estimates; the
harness takes any labeled set via --gold.

===============================================================================
VERBAL (Critical Reasoning + Reading Comprehension) - same eval, section-aware
===============================================================================
The taxonomy tagger, ai_ingest clients, and eval harness are all section-aware
(--section quant|verbal). Verbal adds 16 leaf topics: 9 Critical Reasoning under
gmat::verbal::cr::* (assumption, strengthen, weaken, evaluate, inference,
explain_paradox, flaw, boldface, complete_argument) and 7 Reading Comprehension
under gmat::verbal::rc::* (main_idea, detail, inference, function, structure,
tone, application).

HELD-OUT SET: gmatwiz/content/verbal_gold.json - 39 authored items (27 CR from
verbal_seed.json + 12 RC from verbal_rc_seed.json), each with a human-assigned
gold leaf topic. All content is ORIGINAL (license authored-gmatwiz): the CR
arguments and the RC passages are written for GMATWiz. The prep books provided by
the user were used only to confirm the standard CR/RC question-type taxonomy,
never as a source of text. The ai_ingest leakage-quarantine
(near-duplicate-of-official -> dropped) stays active so any AI-generated item that
reproduces copyrighted source text is removed before admission.

REPRODUCE (verbal):
  cd gmatwiz/content
  python3 make_verbal_rc.py            # (re)build the RC passages + gold
  python3 -c "import json;a=json.load(open('verbal_seed.json'));b=json.load(open('verbal_rc_seed.json'));json.dump(a+b,open('verbal_gold.json','w'),indent=2)"
  # Baselines only (offline) - keyword + vector on the CR+RC gold set:
  python3 eval_tagging.py --section verbal --gold verbal_gold.json \
      --report eval_report_verbal.json
  # Full run incl. the AI tagger (same app key as quant):
  export OPENAI_API_KEY="$(npx -y firebase-tools@latest functions:secrets:access OPENAI_API_KEY --project gmatwiz)"
  python3 eval_tagging.py --section verbal --gold verbal_gold.json \
      --report eval_report_verbal.json

RESULTS - BASELINES (real, this commit; gold=39 [27 CR + 12 RC], cutoff=0.6)
metric                 keyword        vector
accuracy                 92.3%         23.1%
domain-acc               94.9%        100.0%
coverage>=cut            97.4%          0.0%
acc@accepted             94.7%           -
conf-wrong                5.1%          0.0%

Reading it (honest nuance): the KEYWORD baseline is strong on this authored set
(92.3% exact leaf) because CR/RC question types are signalled by stereotyped stem
phrasing (CR: "most seriously weakens", "two portions in boldface"; RC: "primary
purpose", "according to the passage", "the author mentions X in order to") that
regex rules capture well, and the gold was authored in the same house as the
rules. domain-acc (94.9%) is now meaningful: with two verbal domains (cr vs rc)
present, it measures whether an item is at least routed to the right family. The
tf-idf vector baseline is weak (23.1%) and never clears the 0.6 cutoff. The AI row
requires the OpenAI key (offline --mock is random, not a real score); on this
deliberately keyword-friendly gold the bar for "AI beats keyword" is high, so the
honest read is that the keyword rules are the cheap high-precision tagger for
CR/RC while the AI tagger is the coverage/robustness layer for the varied phrasing
that authored gold under-samples.

DATA INSIGHTS (Data Sufficiency + Two-Part + Multi-Source) - same eval
===============================================================================
The taxonomy tagger, ai_ingest clients, and eval harness are section-aware for a
third section too (--section di). Data Insights adds 3 leaf topics under
gmat::di::reasoning::* (data_sufficiency, two_part_analysis, multi_source_reasoning)
- the pragmatic MCQ-compatible scope (Data Sufficiency on the standard 5-option
card, Two-Part Analysis authored MCQ-compatible, Multi-Source Reasoning with its
source stimulus carried in the passage field). Graphics Interpretation + Table
Analysis are deferred (they need an interactive, non-MCQ UI).

HELD-OUT SET: gmatwiz/content/di_seed.json - 12 authored items (4 per leaf), each
with a human-assigned gold leaf topic. All content is ORIGINAL (license
authored-gmatwiz): the DS statement pairs, Two-Part scenarios, and Multi-Source
sources are written for GMATWiz on neutral topics. The official GMAC Data Insights
guides provided by the user were used only to confirm the standard DI question-type
structure, never as a source of text. The ai_ingest leakage-quarantine
(near-duplicate-of-official -> dropped) stays active so any AI-generated DI item
that reproduces copyrighted source text is removed before admission.

REPRODUCE (di):

  python3 eval_tagging.py --section di --gold di_seed.json \
