-------------------------------------------------------------------------------
WHY A SIGKILL CANNOT CORRUPT THE COLLECTION (the claim under test)
-------------------------------------------------------------------------------
The engine opens the DB in WAL journal mode with an exclusive lock
(rslib/src/storage/sqlite.rs: journal_mode=wal, locking_mode=exclusive) and
commits EVERY operation as its own transaction (rslib/src/collection/
transact.rs: savepoint 'rust' -> release). A SIGKILL is a process crash, not
a power loss, so committed WAL frames are already on disk: the next open
replays them (committed reviews survive = no loss) and discards any
half-written transaction (= no corruption). We prove that holds 20x running.

OFFLINE / AI-DEGRADE ARM - scores still compute with AI off
-------------------------------------------------------------------------------
The score engine is pure local Rust+SQL over the revlog/cards
(rslib/src/gmatwiz.rs gmat_scores_json) - it never calls the network or AI,
so pulling the network cannot stop it returning a score. The AI FEATURES
(item generation / error-log coach / 7f checker) are what turn off, and they
degrade CLOSED, not crash:
  * ts/routes/gmat/ai.ts  getAiEnabled(): returns FALSE unless auth+Firebase
    are configured AND the user opted in -> AI is OFF by default, and NO
    network call is attempted when running auth-less/offline.
  * ts/routes/gmat/aiChecker.ts  FAIL_CLOSED: if the AI is unavailable,
    checkItem() returns {pass:false, reasons:['ai_unavailable']} - a
    generated item is rejected rather than admitted unchecked.
So offline: AI features cleanly disable, and the app still shows a score.
