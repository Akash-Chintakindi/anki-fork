WHAT GMATWIZ IS
- A GMAT prep app built INSIDE a fork of Anki (brownfield): we edit Anki's own
  Rust/Python/TypeScript, reusing its spaced-repetition + FSRS engine, and turn
  it into a structured GMAT tutor. Not a layer on top of Anki.
- Current scope: GMAT Focus **Quant, Problem Solving only** (arithmetic +
  algebra; no geometry, no Data Sufficiency). Verbal + Data Insights are NOT in
  yet (post-MVP).

DONE - DESKTOP (runs via ./run, one Anki window)
-------------------------------------------------------------------------------
- Anki builds from source; GMATWiz opens as a screen INSIDE the main window
  (a new "gmat" state in qt/aqt/main.py). The native Anki top-bar tabs
  (Decks/Add/Browse/Stats/Sync) are now HIDDEN (filtered in the
  top_toolbar_did_init_links hook in qt/aqt/gmat.py) - the toolbar shows only
  GMATWiz. Their useful functions are integrated into the GMATWiz UI:
    * Progress tab = GMAT-scoped stats (gmat_stats handler) + "Open full Anki
      stats" (bridges to mw.onStats via taskman.run_on_main).
    * "Free study (all decks)" = bridges to mw.moveToState("deckBrowser") - the
      escape hatch.
    * "Sync" button in the GMATWiz header = bridges to mw.on_sync_button_clicked.
      On first use (no sync configured) it AUTO-LOGS-IN to the self-hosted
      GMATWiz server (defaults match the phone: gmat:wiz @ 127.0.0.1:27811,
      overridable via col config gmatSyncEndpoint/User/Pass) and stores the
      hkey + custom sync URL in the profile, so desktop<->phone sync "just
      works" with no Preferences setup. Existing sync config is respected.
- REQUIRED RUST ENGINE CHANGE: topic-aware scheduling (rslib) - weak-topic cards
  surface first while FSRS intervals stay valid; per-card topic mastery in
  CardData (no schema migration); SetTopicMastery RPC; toggle in deckconfig.
  3 Rust unit tests + 1 Python test; undo + integrity verified.
- Content: GMAT PS notetype + importer (pylib/anki/gmatwiz.py); 273 questions
  (42 authored seed + 231 open-licensed AQuA-RAT) in gmatwiz/content/;
  18 authored lessons in gmatwiz/lessons/.
- App experience (SvelteKit in ts/routes/gmat/, served by qt/aqt/mediasrv.py):
  * Today: an ASSEMBLED, time-budgeted session (due reviews -> next lesson ->
    targeted weak practice) sized to daily minutes, with a dated on/behind-
    track pacing banner to the exam. "Start today" walks the checklist.
  * Diagnostic: 21-question timed Quant pretest -> per-topic mastery ->
    personalized plan (weakest-first); auto-enables topic-aware scheduling.
  * Learn: 18 lessons as I-do -> we-do -> you-do (application-first).
    Completing a lesson SCHEDULES its you-do items as real cards (idempotent),
    so application flows into spaced review.
  * Practice: real review loop on the REAL scheduler (records reviews). Every
    answer EMA-updates that topic's mastery (0.7*old + 0.3*correct) and writes
    it back via set_topic_mastery -> the plan + topic-aware order keep
    re-adapting (a living plan, not diagnose-once).
  * Error Log -> REPAIR LOOP: every miss must be classified in one tap
    (careless / concept gap / timing; corrects can be flagged "I guessed").
    concept_gap -> extra mastery penalty + lesson items scheduled + a "Repair"
    block in Today (cleared by relearning); timing -> topic marked for timed
    drill; guess -> mastery penalty countering the lucky Good. Error Log tab
    has filters + per-entry "Repair now" (relearn or practice).
  * TIMING ANALYTICS: real per-question ms recorded on both platforms (deck
    answer-time cap raised to 300s); the ENGINE computes pace vs the 2:08
    GMAT target and separates fast-but-wrong (careless) from slow-but-correct
    (fragile) in the Performance score. Practice shows a live pace timer.
  * TIMED MOCKS: 21-question / 45:00 adaptive section (harder on right,
    easier on wrong; prefers never-seen questions; sealed until the end).
    Mocks bypass FSRS (no interval distortion): results update topic mastery
    (living plan), misses require classification, and the ENGINE folds mock
    history into Readiness (mock Q history + projection-vs-mock gap warning).
    Today suggests a mock when learning completes or exam <= 21 days (7-day
    cadence). Now EXAM-ACCURATE: flag/bookmark, a pre-submit review grid, and
    up to 3 answer changes in review (GMAT Focus rules); Quant-only for now.
  * OFFICIAL-SCORE CALIBRATION (Readiness tab): log a real practice-test Quant
    score; the desktop snapshots the app's raw projection at that moment
    (gmatOfficialScores config, projected_at_entry). The ENGINE computes a bias
    = mean(official - projected) and shows a CALIBRATED Q as the headline while
    keeping the raw heuristic + residual visible (honesty rule). Confidence
    bumps with >=2 official scores. Shared by desktop + phone (engine-owned).
  * USER-UPLOAD IMPORT (Tools -> "Import my questions..."): bring your own
    question set (JSON in our schema or CSV) to practice in-app - the PRD
    ContentSource user-upload seam. Validates rows (reasons for skips), tags
    source="user-upload" + tag gmatwiz::user-upload, imports into GMAT::Quant.
    Legal: we never bundle copyrighted official tests (PRD sec 19).
  * Readiness dashboard: three SEPARATE scores, each with the honesty rule.
  * Progress tab: focused, GMAT-scoped activity - reviews today, day streak,
    time today, a 7-day review sparkline, the review pipeline (new/learning/
    young/mature), and a 7-day due forecast; computed read-only via find_cards +
    a scoped revlog query. Buttons open Anki's full stats and free-study decks.
- Exponent rendering (x^2) fixed across practice/diagnostic/lessons.

THE THREE SCORES (honesty rule: point + range + evidence + give-up abstention)
- Memory: calibrated from the review log - observed retention + 95% CI, a
  reliability diagram by past-interval bucket vs target, ECE + calibrated badge.
  Abstains under 150 graded reviews.
- Performance: from FIRST-EXPOSURE attempts (new questions), distinct from
  Memory; point + CI, per-topic weak spots, held-out per-topic-model vs
  global-baseline Brier check. Abstains under 50 attempts.
- Readiness: projected GMAT Focus QUANT section score (Q60-90) + range +
  confidence, gated by the full give-up rule (coverage >=50%, >=200 reviews,
  >=50 attempts, ECE <=0.10). Total (205-805) ABSTAINS (needs Verbal + DI).
  Mapping is a documented heuristic; now CALIBRATED against the user's real
  official scores when logged (bias offset, raw + calibrated both shown).

DONE - SYNC (PRD Phase 7) + INSTALLER (PRD Section 17)
-------------------------------------------------------------------------------
- SELF-HOSTED SYNC SERVER: the engine already ships one. Run with
    ./tools/gmat-sync-server.sh     (defaults: gmat:wiz @ 127.0.0.1:27811,
                                     data in out/syncserver)
  NOTE: the server must be RUNNING for sync to work; "Sync failed" on the phone
  almost always means nothing is listening on the endpoint. The app now shows
  the engine's error reason. First/divergent syncs obey the server-dictated
  direction (an empty side can never wipe the populated side).
- DESKTOP sync: Anki's built-in full sync; point at the server via the endpoint.
- MOBILE sync: rslib/src/gmatwiz.rs Backend::gmat_sync + gmat_sync_collection_at;
  FFI gmatwiz_sync(path, endpoint, user, pass, prefer_upload); Swift
  GmatwizEngine.sync + Sync down/up buttons in the app. The phone closes its
  collection (single-writer lock), the engine reopens/syncs, then it reopens.
- PROVEN two-way, both directions: desktop<->desktop AND desktop<->phone (via the
  exact mobile C FFI) converge (revlog counts match) against the self-hosted
  server. iOS app builds + runs in the simulator with the Sync controls.
- CONFLICT RULE:
  * Reviews (revlog) are append-only -> merged as a union; never conflict.
  * Note/card edits: last-writer-wins per object by USN/mtime (Anki's model).
  * First/divergent sync needs a FULL sync; direction is explicit (no silent
    loss): desktop is the authoring source of truth; the phone offers Sync down
    (take server) and Sync up (push phone). Offline edits reconcile on reconnect
    via normal sync; if both diverged at schema level, a full sync in the chosen
    direction resolves it.
- INSTALLER: native macOS .dmg via Briefcase (ad-hoc signed -> runs on a clean
  machine), contains GMATWiz. Build:  ./ninja wheels && ./ninja installer:package
  Output: out/installer/dist/anki-26.05-mac-apple.dmg (~229 MB). Wheels alone:
  out/wheels/anki-*.whl + aqt-*.whl.

HOW TO RUN
-------------------------------------------------------------------------------
Desktop:
  cd /Users/akashchintakindi/Documents/AlphaProjects/GMATWiz/anki-fork
  ./run
  # In app: Tools -> Import GMAT Quant (once), then use the GMATWiz tab.

Mobile (simulator):
  xcodegen generate --spec ios/project.yml
  xcodebuild -project ios/GMATWizPhone.xcodeproj -scheme GMATWizPhone \
    -sdk iphonesimulator -configuration Debug -derivedDataPath ios/build \
    -destination 'platform=iOS Simulator,name=iPhone 17' build
  xcrun simctl boot "iPhone 17"; open -a Simulator
  xcrun simctl install booted ios/build/Build/Products/Debug-iphonesimulator/GMATWizPhone.app
  xcrun simctl launch booted com.gmatwiz.phone
  # Prove reviews are real (quit app first - it holds an exclusive DB lock):
  #   DB="$(xcrun simctl get_app_container booted com.gmatwiz.phone data)/Documents/gmat.anki2"
  #   sqlite3 "$DB" "select count() from revlog;"

Rebuild after Rust FFI changes:
  PROTOC="$(pwd)/out/extracted/protoc/bin/protoc" CARGO_TARGET_DIR=out/rust \
    cargo build -p gmatwiz_ffi --features rustls --target aarch64-apple-ios-sim
  # then recreate out/ios/GmatwizFFI.xcframework (see ios/build-ios.sh).
