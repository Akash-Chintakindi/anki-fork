WHY THESE, AND WHY NOW
- The PRD deliberately shipped with AI OFF (S3 non-goal, S16) but pre-designed
  the seams: a ContentSource adapter, a CardGenerator interface feeding a "7f"
  checker, and a Coach interface for turning error-log reflections into
  takeaways. This pass lights up those seams in dependency order.
- Everything is grounded in the two Brainlift SPOVs (PRD S2):
    SPOV 1 - "Structure is the product" (adaptive learning structure).
    SPOV 2 - "Application over lessons" (lessons create confidence; application
             creates competence) + "error logs are non-negotiable: capture WHY
             a miss happened and what to do next time."
- HARD RULE honored by every feature (PRD S16, S14.4): AI is flag-gated and must
  DEGRADE CLEANLY. If AI is off, unconfigured, offline, times out, or returns
  garbage, the app falls back to its existing non-AI behavior - no crash, no
  blocked review, scores/reviews keep working.

ARCHITECTURE (one AI client, two apps)
- Desktop and iOS render the SAME SvelteKit app, which already initializes
  Firebase (Auth + Firestore). Runtime AI lives in the TypeScript layer and is
  proxied through a Firebase Cloud Function (gmatGenerate) that calls OpenAI, so
  the API key stays server-side and ONE implementation serves BOTH platforms - no
  Python/Rust AI duplication. The Rust/Python backends only store data.
- Content ingestion is the exception: a build-time Python tool (offline), not
  part of the running app.
- PROVIDER NOTE: the runtime seam started on Firebase AI Logic (Gemini) and was
  later re-pointed to OpenAI via the gmatGenerate Cloud Function (see F). The
  ai_ingest.py build tool is provider-flexible (Gemini/OpenAI/heuristic/mock).

-------------------------------------------------------------------------------
F - SHARED AI PLUMBING  (the seam everything else uses)
-------------------------------------------------------------------------------
Files: ts/routes/gmat/ai.ts, ts/routes/gmat/aiChecker.ts, functions/ (proxy)
What:
- ai.ts wraps the AI call behind one function, generateJson<T>(prompt, schema):
  returns { ok:true, value } or { ok:false, reason } and NEVER throws. It calls
  the `gmatGenerate` Firebase callable (Functions v2, us-central1) which holds the
  OPENAI_API_KEY secret and returns structured JSON (schema is normalized to plain
  JSON Schema before sending). Timeout-wrapped; any failure -> {ok:false}.
- functions/ = the Cloud Functions v2 project implementing gmatGenerate
  ({prompt, responseSchema?, model?} -> {text}); default model gpt-4.1-mini
  (overridable), key via Functions secret, App Check toggle (off until clients
  send tokens). Compiles now; deploys at go-live.
- Flag gating: aiEnabled requires Firebase configured AND the user switch ON;
  override via localStorage "gmatwiz.ai" mirrored to synced config gmatAiEnabled.
  DEFAULT OFF (AI needs Blaze + the key). getAiEnabled()/setAiEnabled() exported.
- aiChecker.ts is the PRD S14.8 "7f" quality gate: checkItem() scores an item on
  correctness / in-scope / well-formed / teaching-quality with a pass cutoff,
  FAIL-CLOSED (pass:false) when AI is unavailable. It gates every AI-generated
  question before it is admitted (see the Drill/Study generator below).
Reasoning: PRD S16 "Define clean seams so AI plugs in later; everything runs
with AI off." One seam + the degrade-cleanly guarantee, reused by the Coach, the
Drill/Study generator, and the quiz/test generator.

AI ON/OFF SWITCH  (user control, default OFF)
-------------------------------------------------------------------------------
Files: ts/routes/gmat/ai.ts, GmatApp.svelte (Progress tab), api.ts,
       qt/aqt/mediasrv.py, rslib/src/gmatwiz.rs
A visible toggle in the Progress tab ("AI question generation & coaching") wired
to getAiEnabled/setAiEnabled, mirrored to synced config gmatAiEnabled (in
GMAT_STATE_KEYS on both platforms; synced-wins-on-load). When OFF: no gmatGenerate
calls at all, Drill/Study use only the fixed bank, the Coach button is hidden -
the full degrade path. Default OFF until the user enables it (and Blaze + key set).

-------------------------------------------------------------------------------
HYBRID DRILL / STUDY GENERATION  (application over lessons, SPOV 2)
-------------------------------------------------------------------------------
Files: GmatApp.svelte, api.ts, qt/aqt/mediasrv.py, rslib/src/gmatwiz.rs
Drill serves real FSRS due cards first; when the queue is exhausted (or "Generate
more") AND AI is ON, it generates a small batch of NEW weakest-topic items via
generateJson, gates EACH through aiChecker (drop fails), then admits survivors:
desktop adds them as real "GMAT PS" notes (tagged gmatwiz::ai-generated) so FSRS
tracks them; mobile practices them ephemerally (no rslib importer yet). Study's
per-topic practice generates the same way. Wrong answers -> error log + repair.
An "End session" review shows the right/wrong summary. AI OFF -> fixed bank only.

-------------------------------------------------------------------------------
E - AI CONTENT INGESTION  (build-time content quality)
-------------------------------------------------------------------------------
File: gmatwiz/content/ai_ingest.py   (reuses taxonomy.py; sibling of scraper.py)
What (four passes over a normalized question bank):
  1. Validity/scope filter - drop malformed items; quarantine out-of-scope
     geometry / Data Sufficiency (PRD S5 = Problem Solving, arithmetic+algebra).
  2. AI auto-tag - classify each item to exactly one leaf topic in
     taxonomy.TAXONOMY with a confidence; low-confidence flagged for review.
  3. Semantic dedupe - group reworded-equivalent duplicates (embeddings, or
     token-set similarity offline); near-dups of official/copyrighted items are
     quarantined as leakage risks (PRD S14.3).
  4. Difficulty estimate - re-level easy/medium/hard (the scraped bank skews
     ~80% "medium").
Degrade: client is Gemini (if GEMINI_API_KEY/GOOGLE_API_KEY + SDK present),
else a HeuristicClient (taxonomy keyword tagger + rules), else --mock (offline
deterministic). Results cached in gmatwiz/content/.ai_cache.json (gitignored).
Safety: never overwrites questions.json without --write; writes to
questions.curated.json + a machine-readable report by default.
Run (user step, needs a key):
  cd gmatwiz/content
  python3 ai_ingest.py --in questions.json --out questions.curated.json \
      --report ai_ingest_report.json
  # review low_confidence_items / duplicate_groups / quarantined, then promote.
Offline smoke test (no key): add --mock.
Reasoning: PRD S13 pipeline already calls for "topic auto-tag" and
"near-duplicate detection"; AI raises tagging/dedupe/difficulty quality, which
every downstream feature (coverage map, topic-aware scheduling, future
generation) keys off. Build-time, so runtime degrade rules don't apply.

-------------------------------------------------------------------------------
A - AI ERROR-LOG COACH  (the headline, most Brainlift-aligned feature)
-------------------------------------------------------------------------------
Files: ts/routes/gmat/coach.ts, GmatApp.svelte (Error Log view), api.ts,
       qt/aqt/mediasrv.py, rslib/src/gmatwiz.rs
What:
- On a logged miss, coachMiss(entry) asks Gemini (via F's generateJson) to first
  solve the problem, then return a structured CoachTakeaway:
    { root_cause, rule, check, next_action }
  shown in the Error Log as "Why you missed it / The rule / 10-second check /
  Next". The model explains the KNOWN-correct reasoning (miss context is passed
  in), it does not invent facts.
- Miss context: error-log entries are enriched (best-effort) with the option
  texts + explanation at log time so the coach has full context.
- Persistence + sync: the takeaway is saved on the error entry (ai_takeaway)
  via a new endpoint gmatSetErrorTakeaway, implemented with desktop+mobile
  parity (mediasrv.py + gmatwiz.rs). Because gmatErrorLog is already synced
  state, takeaways cache and follow the account across devices - generated once.
- Degrade: the "Coach this miss" button only shows when AI is enabled; on any
  failure it shows "AI coaching unavailable" and the existing repair loop
  (careless/concept_gap/timing repair + "Repair now") is untouched.
Reasoning: The Brainlift calls error logs non-negotiable and wants the WHY + the
next step (PRD S2, S10). The app already captured a coarse why-tag and scheduled
repairs; the coach turns each miss into a specific, personalized root-cause and
fix - directly serving SPOV 2 (application/repair over passive lessons). PRD S16
pre-named this the "Coach interface."

-------------------------------------------------------------------------------
GUARDRAILS (all features)
-------------------------------------------------------------------------------
- Flag-gated, OFF-safe: builds, typechecks, and runs with AI disabled.
- Degrade cleanly: generateJson never throws; UI hides/greys AI affordances and
  falls back; reviews and scores never depend on AI.
- 7f checker (aiChecker.ts) gates any future AI-GENERATED card, fail-closed.
- Leakage (PRD S14.3): ingestion quarantines near-copies of official items.
- No official/copyrighted content enters the learning pool (PRD S13).
- Honesty rule (PRD S4) still applies to any AI-derived claim.

-------------------------------------------------------------------------------
PREREQUISITE TO GO LIVE (one-time, like publishing the Firestore rules)
-------------------------------------------------------------------------------
- Upgrade the `gmatwiz` Firebase project to the Blaze plan (Cloud Functions).
- Set the OpenAI key + deploy the proxy:
    firebase functions:secrets:set OPENAI_API_KEY        (paste sk-... key)
    cd functions && npm install && npm run build
    firebase deploy --only functions
  (optional: OPENAI_MODEL to change the default; GMAT_ENFORCE_APP_CHECK=true once
  clients send App Check tokens). Until deployed, the AI switch stays off and the
  app uses the fixed bank + non-AI behavior everywhere.
- Then flip the "AI features" switch ON in the Progress tab.
- For the E build tool's live run, set GEMINI_API_KEY/GOOGLE_API_KEY (or OpenAI)
  in the shell; without a key it runs heuristic/mock.
